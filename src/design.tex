\section{\sysname Design}
\label{sec:design}

\subsection{Overview}
\label{sec:design:overview}

\sysname resolves the useful-compute ratio vs.\ throughput tension
(Section~\ref{sec:bg:gap}) through a two-stage design
(Figure~\ref{fig:overview}):

\begin{enumerate}
  \item \textbf{Construction stage.} A lightweight kernel scans the input
        active mask $M_{in}$ and evaluates Eq.~\eqref{eq:mask} for each
        output position, emitting a spatially compact \emph{worklist}
        $\mathbf{q} = (q_1, \ldots, q_m)$ of $m = \|M\|_0$ active output
        indices drawn from $G_{out}$. The worklist exposes tile-local
        spatial locality (Section~\ref{sec:design:construct}).

  \item \textbf{Compute stage.} A convolution kernel consumes $\mathbf{q}$
        as its coordinate source. For each $q_j$, the kernel decodes the
        spatial coordinate, reads the receptive-field input
        $\mathbf{x}[\mathcal{N}(q_j)]$, and accumulates
        Eq.~\eqref{eq:conv} identically to the dense pipeline
        (Section~\ref{sec:design:compute}).
\end{enumerate}

The computed set $\mathcal{C} = \{q_1, \ldots, q_m\}$ contains only active
positions (high useful-compute ratio), while the per-position datapath
remains that of the dense pipeline (near-dense throughput). The
construction stage operates entirely in mask space and its cost is
independent of channel dimensions, making its overhead negligible relative
to the compute stage.

\subsection{Worklist Construction}
\label{sec:design:construct}

The construction stage discovers active coordinates and arranges them to
preserve tile-local spatial locality, with negligible overhead relative to the
compute stage it feeds.
It produces the worklist $\mathbf{q}$ from the input mask $M_{in}$ by
performing a per-tile gather followed by concatenation over the tile
partition $T_1, \ldots, T_N$ defined in Eq.~\eqref{eq:tile}. For each
output position, the stage evaluates Eq.~\eqref{eq:mask} to determine the
output mask $M$ (requiring $k^2$ boolean reads of $M_{in}$ per position),
then gathers active indices within each tile in parallel.

\paragraph{Per-tile gather}
For each tile $T_n$, the stage selects positions where $M$ is active,
yielding a support set:
\begin{equation}
  \mathcal{I}_n = \bigl\{\, k \in \{1, \ldots, t_h t_w\} : M[T_n[k]] = 1 \,\bigr\}.
  \label{eq:support}
\end{equation}
It then gathers the corresponding output indices from $T_n$ in row-major
order:
\begin{equation}
  \mathbf{q}_n = (T_n[k])_{k \in \mathcal{I}_n},
  \label{eq:pertile-gather}
\end{equation}
with length $|\mathbf{q}_n| = |\mathcal{I}_n|$.

\paragraph{Worklist concatenation}
The stage concatenates all per-tile subsequences into the final worklist.
Let $P = (P_1, \ldots, P_N)$ be a permutation of $(1, \ldots, N)$
determined by nondeterministic parallel block scheduling. The stage writes:
\begin{equation}
  \mathbf{q} = \bigl[\;\mathbf{q}_{P_1};\; \mathbf{q}_{P_2};\;
               \cdots;\; \mathbf{q}_{P_N}\;\bigr],
  \label{eq:worklist}
\end{equation}
a contiguous array of $|\mathbf{q}| = \|M\|_0$ active output indices.
Each segment $\mathbf{q}_{P_n}$ preserves its internal row-major order;
only the inter-segment ordering depends on the scheduling permutation $P$.
The above is summarized in Algorithm~\ref{alg:construct}: lines~3--4
compute $M$, lines~5--6 perform the per-tile gather, and line~7
concatenates segments into $\mathbf{q}$.


\paragraph{Cost analysis}
The construction stage evaluates Eq.~\eqref{eq:mask} for all
$H_{out} W_{out}$ output positions, performing $k^2$ mask comparisons
per position. Its total work per layer is
$O(H_{out} W_{out} \cdot k^2)$, independent of $C_{in}$ and $C_{out}$.

To estimate the overhead relative to computation, we model per-layer
latency as $\hat{t} = \hat{t}_{\text{construct}} +
\hat{t}_{\text{compute}}$, assuming each is proportional to its
operation count. Construction performs $H_{out} W_{out} \cdot k^2$
operations; computation performs
$\rho^l \cdot H_{out} W_{out} \cdot k^2 \cdot C_{in} \cdot C_{out}$
FMAs. Under this model the overhead ratio is:
\begin{equation}
  \hat{t}_{\text{construct}} \,/\, \hat{t}_{\text{compute}}
  \;\leq\; 1\,/\,(\rho^l \cdot C_{in} \cdot C_{out}).
  \label{eq:overhead}
\end{equation}
For a layer with $C_{in} = C_{out} = 64$ and $\rho^l = 5\%$, this gives
${\approx}\,0.5\%$; at $\rho^l = 1\%$ it is ${\approx}\,2.4\%$.
Construction overhead is thus negligible, and the end-to-end latency of
\sysname is dominated by the compute stage.
Section~\ref{sec:eval} confirms that measured overhead is consistent with
this bound.

\begin{algorithm}[t]
\caption{Worklist Construction}
\label{alg:construct}
\begin{algorithmic}[1]
\REQUIRE Tiles $T_1, \ldots, T_N$, input mask $M_{in}$
\ENSURE Worklist $\mathbf{q}$, output mask $M$
\FOR{each tile $T_n$ \textbf{in parallel}}
  \FOR{each $k \in \{1, \ldots, t_h t_w\}$}
    \STATE $p \leftarrow$ spatial position of $T_n[k]$
    \STATE $M[p] \leftarrow$ Eq.~\eqref{eq:mask}
      \hfill $\triangleright$ test $\mathcal{N}(p)$ against $M_{in}$
  \ENDFOR
  \STATE $\mathcal{I}_n \leftarrow \{k : M[T_n[k]] = 1\}$
  \STATE $\mathbf{q}_n \leftarrow (T_n[k])_{k \in \mathcal{I}_n}$
\ENDFOR
\STATE $\mathbf{q} \leftarrow (\mathbf{q}_{P_1}, \ldots, \mathbf{q}_{P_N})$
  \hfill $\triangleright$ $P$ from block scheduling
\end{algorithmic}
\end{algorithm}


\subsection{Worklist-driven Convolution}
\label{sec:design:compute}

The compute stage fuses position-level selectivity into the dense
pipeline's coordinate-driven execution: it computes Eq.~\eqref{eq:conv}
for exactly the $\|M\|_0$ worklist entries while preserving the
per-position datapath unchanged, summarized as
Algorithm~\ref{alg:compute}.

Following standard implicit GEMM, the workload is partitioned into compute
tiles: the worklist positions are divided into segments of size $B_M$
indexed by $b_m = 1, \ldots, \lceil \|M\|_0 / B_M \rceil$, and the
$C_{out}$ output channels into segments of size $B_N$ indexed by
$b_n = 1, \ldots, \lceil C_{out} / B_N \rceil$; all compute tiles execute
in parallel (line~1 in Algorithm~\ref{alg:compute}). Each compute tile first loads its $B_M$ worklist
entries into $\mathbf{q}'$ (line~2). After initializing the accumulator
(line~3), the compute tile iterates over the $k^2$ receptive-field offsets
(line~4) and within each offset over $C_{in}$ in steps of $B_K$ (line~5):
at each step, the kernel gathers the input slice $\mathbf{x}'$ at offset
$i$ from positions $\mathcal{N}_i(\mathbf{q}')$, loads the corresponding
weight slice $\mathbf{w}'$ from $\mathbf{w}_i$, and accumulates the
partial product (lines~6--8).
Finally, the compute tile writes the accumulated result back to $\mathbf{y}$ at
the worklist addresses (line~9).
% Compared with dense implicit GEMM, the differences are: (1)~each compute
% tile loads its coordinates from the worklist rather than generating them
% sequentially, (2)~the input gather reads from worklist-derived positions
% $\mathcal{N}_i(\mathbf{q}')$, and (3)~the output writeback addresses are
% the worklist entries themselves.

\paragraph{Locality}
Each compute tile processes $B_M$ consecutive worklist entries. Because
construction compacts active positions per tile, consecutive entries within
each $\mathbf{q}_{P_n}$ are row-major neighbors in $G_{out}$ whose
receptive fields overlap in contiguous memory. Locality can degrade across
segment boundaries when active positions are sparse across tiles, but this
construction avoids a fully arbitrary active-position order without requiring a
global sort. Accordingly, \sysname claims tile-local locality rather than
preservation of the global dense raster order.

\begin{algorithm}[t]
\caption{Worklist-driven Convolution}
\label{alg:compute}
\begin{algorithmic}[1]
\REQUIRE Worklist $\mathbf{q}$, input $\mathbf{x}$, weight $\mathbf{w}$
\ENSURE Output $\mathbf{y}$ at positions in $\mathbf{q}$
\FOR{each compute tile $(b_m, b_n)$ \textbf{in parallel}}
  \STATE $\mathbf{q}' \leftarrow \mathbf{q}[\,(b_m{-}1)B_M : b_m B_M\,]$
  \STATE $\mathbf{y}' \leftarrow \mathbf{0}_{B_M \times B_N}$
  \FOR{$i = 1, \ldots, k^2$}
    \FOR{$j = 1, \ldots, \lceil C_{in} / B_K \rceil$}
      \STATE $\mathbf{x}' \leftarrow \mathbf{x}\bigl[\,\mathcal{N}_i(\mathbf{q}'),\; (j{-}1)B_K : jB_K\,\bigr]$
      \\\hfill $\triangleright$ gather input slice
      \STATE $\mathbf{w}' \leftarrow \mathbf{w}_i\bigl[\,(b_n{-}1)B_N : b_n B_N,\; (j{-}1)B_K : jB_K\,\bigr]$
      \\\hfill $\triangleright$ load weight slice
      \STATE $\mathbf{y}' \leftarrow \mathbf{y}' + \mathbf{x}' \cdot \mathbf{w}'^\top$
    \ENDFOR
  \ENDFOR
  \STATE $\mathbf{y}[\mathbf{q}'] \leftarrow \mathbf{y}'$
\ENDFOR
\end{algorithmic}
\end{algorithm}
